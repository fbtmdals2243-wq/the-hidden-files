# CASE_ENGINE.md

# British Ministry of Magic Portal
## Case Engine Design Document

---

# Vision

Cases are not levels.

Cases are official Ministry records.

The player does not "play a mission."

The player reviews, investigates, restores, and reports on Ministry case files.

---

# Core Philosophy

Every case should feel like a government archive document.

The user is an officer with clearance.

Access is earned through rank, department, and trust.

---

# Case Structure

Every case should contain:

- Case ID
- Case Title
- Classification Level
- Status
- Assigned Department
- Lead Officer
- Summary
- Evidence
- Witness Statements
- Timeline
- Internal Notes
- Final Report

---

# Case Status

Possible statuses:

- Locked
- Available
- Active
- Under Review
- Solved
- Sealed
- Reopened

---

# Case Flow

1. Case appears in Archive Cabinet.
2. Player opens Case File.
3. Player reviews summary.
4. Player examines evidence.
5. Player reads witness statements.
6. Player checks timeline.
7. Player submits internal report.
8. Case status changes.
9. NPCs react.
10. Player record is updated.

---

# Clearance System

Level I:
- Training cases
- Minor archive incidents

Level II:
- Corrupted records
- Historical evidence

Level III:
- Memory-related cases
- Restricted personnel files

Level IV:
- Internal Affairs linked cases
- sealed archive material

Level V:
- CASE-ZERO
- CASE-OMEGA
- Ministry-level classified incidents

---

# First Case

## CASE-000
### The Missing Owl

Purpose:
Teach the player how cases work.

Tone:
Small, quiet, slightly strange.

Core Mystery:
A Ministry owl returned without its message.

But the message was already marked as delivered.

---

# Key Principle

The case system should never feel like a quest list.

It should feel like opening restricted Ministry records.
